# Product Spec

## 1. Product Vision

AI Reading Assistant is a cross-device reading app for PDF and EPUB files. It does not primarily summarize whole books. Instead, it helps readers understand the current page, paragraph, selected phrase, named entity, and concept while reading.

The key interaction is contextual annotation:
- The reader sees a page.
- The app extracts visible text or selected text.
- A server-side agent reads product skills and, in v0.1, optional read-only memory/book context files if present.
- The agent returns structured annotations. In the job-based flow, the reader can keep reading while the server continues generation in the background.
- The UI renders those annotations as a sidebar on wide screens or as a dismissible bubble/bottom sheet on small screens.

## 2. Target Platforms

Primary:
- iPhone Safari/PWA
- iPad Safari/PWA
- Android Chrome/PWA

Secondary:
- Desktop browser
- Future Capacitor iOS/Android apps

## 3. Core User Stories

### US-001: Open a book
As a user, I can add a PDF file so that I can read it in the app.

Acceptance criteria:
- v0.1: User can select `.pdf` files.
- v0.2: User can select `.epub` files.
- File metadata is stored locally.
- Last opened book appears in the library.

### US-002: Read a book
As a user, I can read the selected book with basic navigation.

Acceptance criteria:
- v0.1: PDF supports page navigation.
- v0.2: EPUB supports chapter/spine navigation.
- Reading position is restored.

### US-003: Ask about visible content
As a user, I can request AI annotations for the current visible content.

Acceptance criteria:
- v0.1: Current PDF page text is extracted.
- v0.2: Current EPUB section text is extracted.
- Request is sent to the server.
- Response is displayed without blocking reading.

### US-004: Ask about selected text
As a user, I can select text and ask for an explanation.

Acceptance criteria:
- Selected text is captured.
- Surrounding context is included when available.
- Response identifies whether the selected text is a term, person, place, technology, claim, or unknown.

### US-005: Responsive annotation UI
As a user, I get an annotation UI appropriate to my device.

Acceptance criteria:
- >= 1024px: Collapsible right sidebar. Defaults to open on load, but can be closed using an 'X' button to let the reader take 100% width, and reopened via a floating Bot action button.
- 600-1023px: Collapsible side panel.
- < 600px: Floating action button + bottom sheet / bubble.

### US-006: Page-based reading model
As a user, I can read EPUB content in discrete pages without continuous scrolling.

Acceptance criteria:
- A page is defined by the current viewport size, font size, line height, margins, and other pagination settings.
- The reader must not require continuous vertical scrolling for normal reading.
- If a book section is longer than one page, it is split across multiple pages.
- If a page does not fit all content, the overflow must continue on the next page rather than remain hidden in the current page.
- Page count may change when viewport size, orientation, font size, or layout settings change.
- Page labels are shown as `P.X / N`, where `X` is the current page in the current pagination state and `N` is the total number of pages in that same pagination state.
- Chapter and section boundaries may influence page breaks, but they do not replace page-based pagination.

## 4. Functional Requirements

### Reading
- PDF rendering via PDF.js or react-pdf.
- EPUB rendering via epub.js in v0.2.
- Extract text from current PDF page.
- Extract current EPUB chapter/section text in v0.2.
- Track current location.
- EPUB pagination must be page-based, not continuous-scroll based, and must respect the current viewport and font settings.
- The reader may use chapter or section boundaries as navigation targets, but the visible unit presented to the user is the current page.
- Fullscreen (Zen/Focus) mode to hide the application header and eliminate outer margins and borders for maximum reading area, particularly on mobile viewports. Native Fullscreen API is used when supported, with a state-based Zen Mode fallback for unsupported clients.

### AI
- Support OpenAI-compatible Chat Completions API.
- Configurable base URL, model, temperature, and provider credentials according to the deployment mode.
- Server-side prompt assembly.
- Server-side skill loading.
- Structured JSON output.
- Cache AI results by bookId + location + visibleTextHash + selectedTextHash + skillId + skillVersion + model + language + detailLevel + promptVersion + memoryVersion + bookContextVersion.

### Agent Context Files
- `skills/*.md`: behavior modules.
- v0.1: `memory/users/{userId}/MEMORY.md` and `memory/books/{bookId}/BOOK_CONTEXT.md` may be read if present, but are not edited or generated by the annotation flow.
- v0.4+: user memory display and approved updates.
- v0.5+: book context display, approved updates, and recurring-term accumulation.

### Storage
MVP should use browser-local storage:
- IndexedDB for book files, metadata, annotations, reading progress.
- LocalStorage only for small UI/config values.

Server storage:
- SQLite for skill metadata, request logs, cache, user memory if multi-user mode is enabled.
- File system for Markdown skill/memory files in local-first mode.
- Default server cache must not persist raw page text or selected text. Store text hashes and metadata unless the user explicitly enables debug logging.

Identity and storage ownership:
- Client generates `bookId` when a file is imported and sends that opaque ID with annotation requests.
- Server must treat `bookId`, `userId`, and `skillId` as opaque identifiers, not as raw file paths.
- Server must resolve skill, memory, and book-context files from allowlisted base directories after validating identifiers against a conservative pattern such as `[a-zA-Z0-9_-]+`.
- v0.1 server should not create or mutate memory/book-context files during annotation requests. Lazy creation from templates belongs to the later book-context milestone.

## 5. Non-functional Requirements

- Mobile-first performance.
- Do not auto-call LLM on every page turn by default.
- Explicit user action should trigger AI in v0.1.
- Never send full books to the LLM for one annotation request in MVP.
- Return graceful errors for API failures.
- Design so that private/local LLM endpoints can be used.

## 6. Privacy Requirements

- Make it explicit what text is sent to the LLM provider.
- Make it explicit whether request text is retained by the server. Default retention should be response JSON plus request hashes/metadata, not raw book text.
- Support local/private endpoints.
- Do not upload book files to remote storage in MVP unless user explicitly enables sync.
- Store API keys securely enough for the target deployment. For pure PWA, prefer server-side key storage rather than client-side OpenAI keys.

API key handling by deployment mode:
- Local server mode: store provider API keys in server environment variables or a local server-side secret store.
- Hosted server mode: store user-provided provider keys as encrypted server-side secrets, or use an operator-managed key with explicit user disclosure.
- Direct browser-to-provider mode is not supported in v0.1. Local/private LLM endpoints must still be reached through the server agent.

## 7. Risks

- EPUB visible text extraction can be ambiguous due to reflow.
- PDF text extraction may fail on scanned PDFs.
- iOS Safari PWA storage behavior may require careful testing.
- LLM JSON output can be invalid; schema validation and repair are required.
- Prompt injection can appear inside book text or metadata; the server prompt must treat extracted book content as untrusted quoted source material.
